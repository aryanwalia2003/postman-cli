# Performance Profiling Report - March 26, 2026

## Executive Summary
A follow-up performance audit was conducted on the ReqX CLI on **Linux** (replacing previous Windows-based tests). The test pushed the runner to **300 concurrent workers** at a target of **50 RPS**. While success rates remained at 100%, achieved throughput peaked at **18.7 RPS**, confirming a client-side bottleneck. The audit pinpointed high resource contention in the Socket.IO background listeners and significant memory allocation pressure from redundant string conversions.

---

##  CPU Performance Hotspots
| Component | CPU Usage (Cum%) | Status | Root Cause Analysis |
| :--- | :--- | :--- | :--- |
| **Syscalls** | 37.61% (Flat) | **Stable** | Bulk of CPU spent on network syscalls (Read/Write) at 50 VU. |
| **I/O Reading** | 34.86% (Cum) | --- | Overhead of `bytes.(*Buffer).ReadFrom` during response capture. |
| **WebSocket Read** | < 1% (Flat) | **Verified** | Zero-alloc architecture remains effective under 50 VU / 30 RPS. |
| **Terminal I/O** | 0% | **Fixed** | Auto-Quiet mode eliminated console bottleneck. |

**Key Insight:** The shift to 300+ Workers has moved the bottleneck from TLS Handshaking (fixed on March 25) to **Concurrency Coordination** and **I/O Throughput**.
**Optimization:** Implementing "Quiet Mode" by default for high-load tests and optimizing the Socket.IO background reader to avoid string conversions per packet.

---

##  Memory Allocation Bottlenecks
| Component | Allocation % | Status | Root Cause Analysis |
| :--- | :--- | :--- | :--- |
| **Thread Creation** | 34.86% | **Targeted** | OS threads for blocked readers. Reduced from 48% at lower scale. |
| **Buffer Growth** | 13.98% | **Stable** | Minor growth cycles remaining for payloads > 8 KB. |
| **JS Scripting** | 12.02% | --- | Overhead of `goja` runtime for test assertions. |
| **I/O Buffering** | ~14% | --- | Standard `bufio` overhead for active connections. |

###  Comparison with March 25
- **GC Pressure**: Much lower than March 25 thanks to `gjson` integration.
- **New Pressure Point**: The sheer number of active connections/goroutines is now the primary limiting factor for RPS, rather than the data parsing logic itself.

---

##  Results (March 26)
1. **[DONE] Byte-First Architecture:** Migrated `SocketIOExecutor` to `gjson.GetBytes`. Eliminated parsing hotspots.
2. **[DONE] Auto-Quiet Mode:** Terminal bottleneck eliminated for high-load runs.
3. **[DONE] Body Buffer Warm-up:** 8 KB pre-allocation eliminated `growSlice` cycles.
4. **[DONE] Concurrency Throttle:** `pauseCh` mechanism implemented to quiesce idle workers.

**Next Audit (Phase 4):** Focus on reducing `runtime.allocm` by pooling or multiplexing Socket.IO readers at 500+ VU scale, and consider adaptive buffer sizing for large responses.
