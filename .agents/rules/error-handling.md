You are going to design and implement a **production-grade error handling system for a backend service written in Go**. This system is not meant to be a simple helper library or a small utility. The goal is to build a **robust, scalable, developer-friendly error architecture** that can be used consistently across the entire codebase as the system grows. Error handling should be treated as a core architectural component of the backend rather than scattered `if err != nil` checks. The system must make it extremely easy for developers to create, propagate, classify, log, and respond to errors in a consistent and structured way.

The primary design goal is **consistency across the entire application**. Every error that leaves the service must follow the same structure and behavior. This means errors must be machine readable, predictable, and structured. Instead of returning arbitrary error messages, each error should carry structured information such as a stable error code, a safe message that can be returned to clients, and optional metadata for debugging. The internal error should also preserve the original cause so developers can trace the root issue. This separation between **internal debugging information and client-safe responses** is extremely important.

Another important design goal is **excellent developer experience**. Developers using this system should not need to manually construct complex error objects or remember boilerplate patterns. Creating errors should feel simple and expressive. Ideally, developers should be able to write things like `errs.NotFound("user not found")` or `errs.Wrap(err, errs.Database)` and have the system automatically attach context such as stack traces, metadata, and classification. The system should reduce friction, not add complexity. If error creation feels heavy or verbose, developers will avoid using it.

The system must also support **error propagation with context**. In a typical backend architecture, errors travel through multiple layers such as repository (database layer), service (business logic), and handlers (HTTP layer). When an error moves upward through these layers, each layer should have the ability to add contextual information while preserving the original cause. This allows logs to show a full chain of failure instead of a single vague message. For example, a database timeout error might be wrapped by the repository as “failed to fetch user from database,” and then wrapped again by the service layer as “get user service failed.” This error chain dramatically improves debugging in production systems.

The error system must also support **error classification**. Every error should belong to a defined category or kind. These kinds should represent common backend failure scenarios such as internal errors, invalid input, not found errors, unauthorized access, forbidden operations, conflicts, database failures, and external service failures. These error kinds allow the system to automatically determine things like HTTP status codes, retry behavior, and monitoring classification. Using typed error categories instead of random strings makes the system much easier to reason about.

Another requirement is that the system must integrate with **HTTP response handling** in a centralized way. API handlers should not manually format error responses. Instead, there should be a single component responsible for translating internal errors into API responses. This component should inspect the error, determine its type or capabilities, and return a consistent JSON structure to the client. The JSON format should remain stable across the entire API so that frontend clients and external integrations can rely on predictable error responses.

The error system must also support **error wrapping while preserving the original cause**. Go provides native error wrapping mechanisms that allow an error to contain another error while still being discoverable through functions like `errors.Is` and `errors.As`. Your system should fully support this behavior. Any wrapped error should still allow developers to detect the underlying root cause. This makes it possible to handle specific errors such as database “no rows found” conditions without losing higher level context.

Another critical capability is **stack trace capture**. Whenever a new application error is created, the system should capture a stack trace so that developers can identify exactly where the error originated. Stack traces are extremely valuable when debugging complex distributed systems where logs may be the only source of truth. The stack trace should be available internally for logging and debugging but should never be exposed to API clients.

The system should also support **structured metadata** attached to errors. Sometimes additional context is necessary to understand a failure, such as a user ID, request ID, or external service identifier. The error system should allow attaching arbitrary key-value metadata so that logs and observability tools can include rich debugging information. This metadata should be optional and should not make error creation cumbersome.

Another requirement is **panic recovery integration**. Backend servers must never crash due to unhandled panics inside request handlers. The error system should include a panic recovery mechanism that catches panics, converts them into internal errors, logs the relevant information, and returns a safe error response to the client. This ensures that unexpected failures do not bring down the entire server process.

The design should also emphasize **extensibility through interfaces rather than rigid implementations**. Instead of forcing every error to be a specific struct type, the system should define interfaces representing error capabilities such as providing an error code, returning a safe message, or specifying an HTTP status. This allows external libraries or future components to integrate with the error system without tightly coupling everything to one concrete type.

File organization should follow **clear single-responsibility design**. Different concerns such as error types, constructors, HTTP translation, stack trace handling, metadata handling, and panic recovery should live in separate files within an internal error package. File names should be descriptive and reflect their purpose. The goal is to keep each file focused and maintainable while still forming a cohesive system.

Finally, the resulting error handling system should feel like a **small internal framework for failures**. It should encourage correct patterns by default, minimize repetitive code, and make debugging production issues easier. Developers interacting with the system should naturally follow best practices simply by using the provided helpers and abstractions. The end result should be an error architecture that scales with the codebase, improves observability, and provides a consistent developer experience across the entire backend.
